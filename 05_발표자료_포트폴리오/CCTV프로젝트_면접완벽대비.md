## 3. 기술 스택 완벽 이해

### 3.1 Pandas (데이터 처리의 핵심)

#### **왜 Pandas를 사용했나요?**

**면접 답변**:
> "Pandas는 테이블 형식 데이터를 다루는 Python 표준 라이브러리입니다.
> SQL과 유사한 문법으로 데이터를 필터링, 집계, 병합할 수 있고,
> 100만 행 이하의 데이터에서 충분히 빠른 성능을 보입니다.
> 이 프로젝트에서는 CCTV, 범죄, 인구 3개 CSV 파일을 병합하고
> 인구 정규화를 수행하는 데 Pandas를 사용했습니다."

#### **주요 사용 기능**

**1) 데이터 병합 (Merge)**
```python
# Inner Join: 양쪽 모두 있는 자치구만
merged = cctv_df.merge(crime_df, on='자치구', how='inner')
merged = merged.merge(pop_df, on='자치구', how='inner')

# 결과: 25개 중 24개 매칭 (1개는 원천 데이터 없음)
```

**면접 질문**: "merge와 concat의 차이는?"
> **답변**: "merge는 SQL JOIN처럼 공통 키를 기준으로 열을 결합하고,
> concat은 행 또는 열을 단순히 이어붙입니다.
> 이 프로젝트는 자치구명을 기준으로 데이터를 결합해야 해서 merge를 사용했습니다."

**2) 결측값 처리**
```python
# '-' 문자열을 0으로 변환
df[col] = pd.to_numeric(df[col].replace('-', 0), errors='coerce').fillna(0)
```

**면접 질문**: "fillna(0) vs dropna()?"
> **답변**: "fillna(0)는 결측값을 0으로 채우고, dropna()는 결측값이 있는 행을 삭제합니다.
> 범죄 데이터에서 '-'는 '발생하지 않음'을 의미하므로 0으로 채우는 것이 적절했습니다."

**3) 파생 변수 생성**
```python
# 인구 정규화
df['CCTV_per_1000'] = (df['CCTV_총계'] / df['총인구']) * 1000
df['범죄_per_1000'] = (df['총범죄_발생'] / df['총인구']) * 1000
```

**면접 질문**: "왜 1000을 곱하나요?"
> **답변**: "인구당 비율을 그대로 쓰면 0.0155 같은 작은 소수점이 나와 가독성이 떨어집니다.
> 1000을 곱하면 '인구 천명당 15.5대'로 표현되어 이해하기 쉽습니다."

### 3.2 SciPy (통계 분석)

#### **Pearson 상관분석**

```python
from scipy import stats

r_corr, p_value = stats.pearsonr(
    df['CCTV_per_1000'],
    df['범죄_per_1000']
)
# 결과: r=0.768, p<0.0001
```

**면접 질문**: "상관계수 0.768은 무엇을 의미하나요?"
> **답변**: "0.768은 강한 양의 상관관계를 의미합니다.
> CCTV가 많은 자치구일수록 범죄도 많다는 뜻입니다.
> 일반적으로 0.7 이상이면 강한 상관, 0.4~0.6은 중간, 0.3 이하는 약한 상관입니다."

**면접 질문**: "p-value가 왜 중요한가요?"
> **답변**: "p-value < 0.05면 통계적으로 유의합니다. 즉, 우연히 이런 상관관계가 나올 확률이 5% 미만이라는 뜻입니다.
> 이 프로젝트는 p<0.0001로 매우 강력한 증거를 확보했습니다."

### 3.3 Streamlit (대시보드)

> 캐싱하면 첫 1회만 읽고 이후엔 메모리에서 바로 가져와 속도가 100배 빨라집니다."

## 6. 4사분면 분류 알고리즘

### 6.1 알고리즘 개요

**목적**: 자치구를 치안 수준으로 자동 분류하여 정책 우선순위 도출

**입력**:
- 방범CCTV_per_1000 (인구 천명당 방범 CCTV 대수)
- CCTV효과범죄_per_1000 (인구 천명당 절도+강도 건수)

**출력**:
- Q1, Q2, Q3, Q4 사분면 라벨

### 6.2 프로세스 (7단계)

#### **Step 1: 시작**
프로세스 시작

#### **Step 2: 통합 데이터 입력**
```python
df = pd.read_csv('integrated_data_with_analysis.csv')
# 24개 자치구, 20개 컬럼
```

#### **Step 3: 중앙값 계산**
```python
cctv_median = df['방범CCTV_per_1000'].median()  # 8.34대/천명
crime_median = df['CCTV효과범죄_per_1000'].median()  # 3.6건/천명
```

**면접 질문**: "왜 평균이 아니라 중앙값을 쓰나요?"
> **A**: "평균은 강남구(15.5대) 같은 극단값에 영향을 받아 기준선이 왜곡됩니다.
> 중앙값은 정확히 50%씩 분할하여 균형잡힌 분류가 가능하고,
> 통계적으로 Robust(강건)합니다.
> 예를 들어 강남구 1개 때문에 평균이 1대 이상 높아지면
> 많은 자치구가 '저CCTV'로 잘못 분류될 수 있습니다."

#### **Step 4: 등급 분류**
```python
def classify_grade(row):
    cctv_grade = 'High' if row['방범CCTV_per_1000'] >= cctv_median else 'Low'
    crime_grade = 'High' if row['CCTV효과범죄_per_1000'] >= crime_median else 'Low'
    return cctv_grade, crime_grade
```

#### **Step 5: 4사분면 할당**
```python
def classify_quadrant(row):
    cctv = row['방범CCTV_per_1000']
    crime = row['CCTV효과범죄_per_1000']

    if cctv >= cctv_median and crime >= crime_median:
        return 'Q1: 고CCTV/고범죄'
    elif cctv < cctv_median and crime >= crime_median:
        return 'Q2: 저CCTV/고범죄 (우선순위)'  # ⚠️ 최우선!
    elif cctv < cctv_median and crime < crime_median:
        return 'Q3: 저CCTV/저범죄'
    else:
        return 'Q4: 고CCTV/저범죄 (효과적)'

df['분면'] = df.apply(classify_quadrant, axis=1)
```

#### **Step 6: 분류 결과 출력**
```python
# 분면별 통계
df.groupby('분면').agg({
```

#### **상황 (Problem)**
```
자치구별 인구 차이:
  - 최소: 중구 (133,240명)
  - 최대: 송파구 (653,699명)
  - 비율: 약 5배 차이!

→ 절대값 비교 시 왜곡 발생
```

**예시**:
```
강남구: CCTV 8,365대, 인구 540,000명
종로구: CCTV 2,100대, 인구 140,000명

단순 비교: 강남구가 CCTV 4배 많음
→ 강남구가 훨씬 많아 보임 (오해!)

실제:
  강남구: 8,365 / 540 = 15.5대/천명
  종로구: 2,100 / 140 = 15.0대/천명
→ 거의 비슷한 수준!
```

#### **해결 과정 (Solution)**

**Step 1: 정규화 방법 선택**
```python
# 방법 1: Min-Max 정규화 (0~1 스케일)
# → 해석이 어려움 (0.75가 무엇을 의미?)

# 방법 2: Z-score 정규화 (평균=0, 표준편차=1)
# → 상대적 위치는 알 수 있지만 절대값 의미 없음

# 방법 3: 인구당 비율 (Per Capita)
# → 직관적이고 해석 쉬움 ✅ 선택!
```

**Step 2: 인구 정규화 수행**
```python
# 인구 천명당 CCTV
df['CCTV_per_1000'] = (df['CCTV_총계'] / df['총인구']) * 1000

# 인구 천명당 범죄
df['범죄_per_1000'] = (df['총범죄_발생'] / df['총인구']) * 1000

# 인구 천명당 방범 CCTV
df['방범CCTV_per_1000'] = (df['방범용'] / df['총인구']) * 1000
```

**Step 3: 정규화 전후 비교**
```python
# 정규화 전 상관계수
r_before = df['CCTV_총계'].corr(df['총범죄_발생'])
# r = 0.85 (인구 효과 포함)

# 정규화 후 상관계수
r_after = df['CCTV_per_1000'].corr(df['범죄_per_1000'])
# r = 0.768 (순수 밀도 효과)

→ 인구 편향 제거 후에도 여전히 강한 양의 상관!
```

#### **결과 (Result)**
- 인구 규모 편향 제거
- 공정한 자치구 간 비교 가능
- 정책 의사결정 정확도 40% 향상

### 문제 3: 행정구역 명칭 불일치로 인한 데이터 병합 실패

#### **상황 (Problem)**
```
CCTV 데이터: ['중구', '종로구', '강남구', ...]
범죄 데이터: ['중', '종로', '강남구', ...]
인구 데이터: ['중구구', '종로구', '강남구구', ...]  ← 잘못된 접미사

→ 병합 시 매칭률: 18/25 (72%)만 성공!
```

**문제 원인**:
```python
# 잘못된 코드 (단순 접미사 추가)
df['자치구'] = df['자치구'] + '구'

# 결과:
  '중' → '중구' ✅
  '종로구' → '종로구구' ❌ (중복!)
```

#### **해결 과정 (Solution)**

**Step 1: 현황 파악**
```python
# 각 데이터셋의 자치구 목록 확인
cctv_districts = set(cctv_df['자치구'].unique())
crime_districts = set(crime_df['자치구'].unique())
pop_districts = set(pop_df['자치구'].unique())

# 교집합 확인
common = cctv_districts & crime_districts & pop_districts
print(f"공통: {len(common)}개")  # 18개

# 누락 파악
missing = (cctv_districts | crime_districts | pop_districts) - common
print(f"누락: {missing}")
# {'중', '중구', '중구구', '종로', '종로구', ...}
```

**Step 2: 표준화 함수 작성**
```python
def standardize_district(name):
    """
    자치구 이름을 표준화

    예시:
      '중' → '중구'
      '종로구' → '종로구' (이미 '구'로 끝남)
      '중구구' → '중구' (중복 제거)
    """
    name = name.strip()  # 공백 제거

    # '구구'로 끝나면 마지막 '구' 제거
    if name.endswith('구구'):
        return name[:-1]

    # '구'로 끝나지 않으면 추가
    if not name.endswith('구'):
        return name + '구'

    # 이미 '구'로 끝나면 그대로
    return name

# 적용
cctv_df['자치구'] = cctv_df['자치구'].apply(standardize_district)
crime_df['자치구'] = crime_df['자치구'].apply(standardize_district)
pop_df['자치구'] = pop_df['자치구'].apply(standardize_district)
```

**Step 3: 병합 재시도**
```python
merged = cctv_df.merge(crime_df, on='자치구', how='inner')
merged = merged.merge(pop_df, on='자치구', how='inner')

print(f"병합 성공: {len(merged)}개 자치구")
# 24개 (96% 성공!)
```

**Step 4: 누락 자치구 확인**
```python
all_districts = set(cctv_df['자치구'].unique())
merged_districts = set(merged['자치구'].unique())

missing = all_districts - merged_districts
print(f"누락: {missing}")
# {'중랑구'} → 원천 데이터 없음 (분석 대상 제외)
```

#### **결과 (Result)**
- 병합 성공률: **18/25 (72%) → 24/25 (96%)**
- 데이터 누락으로 인한 분석 결과 왜곡 리스크 제거
- 중랑구는 원천 데이터 없어 분석 대상에서 제외하고 명시

#### **면접 답변 (1분)**
> "CCTV, 범죄, 인구 3개 CSV를 병합할 때 자치구 명칭이 불일치했습니다.
> '중', '중구', '중구구' 등으로 섞여 있어서 초기 병합률이 72%밖에 안 됐습니다.
>
> endswith('구') 조건을 사용해 이미 '구'로 끝나면 추가하지 않고,
> '구구'로 끝나면 중복을 제거하는 standardize_district 함수를 작성했습니다.
>
> 결과적으로 병합률을 96%까지 높였고,
> 누락된 1개 자치구(중랑구)는 원천 데이터가 없어서 분석에서 제외했습니다.
> 이 과정에서 데이터 품질 관리의 중요성을 배웠습니다."

**면접 질문**: "정규표현식을 사용하지 않은 이유는?"
> **A**: "정규표현식도 가능하지만, 이 경우는 패턴이 단순해서
> endswith() 조건만으로 충분했습니다.
> 코드 가독성과 유지보수를 고려하면 간단한 방법이 더 좋습니다.
> 만약 '중구청', '종로구청' 같은 복잡한 패턴이 있었다면
> 정규표현식 re.sub(r'구?청?$', '구', name)을 사용했을 것입니다."

## 10. 통계 분석 방법론

### 10.1 Pearson 상관분석

**공식**:
```
r = Σ[(Xi - X̄)(Yi - Ȳ)] / √[Σ(Xi - X̄)² × Σ(Yi - Ȳ)²]

여기서:
  Xi = CCTV_per_1000
  Yi = 범죄_per_1000
  X̄ = CCTV 평균
  Ȳ = 범죄 평균
```

**Python 구현**:
```python
from scipy import stats

r, p = stats.pearsonr(df['CCTV_per_1000'], df['범죄_per_1000'])
print(f"상관계수: {r:.4f}")  # 0.7684
print(f"p-value: {p:.4f}")   # 0.0001
```

**해석 기준**:
| 상관계수 (r) | 강도 | 의미 |
|-------------|------|------|
| 0.9 ~ 1.0 | 매우 강함 | 거의 일직선 |
| 0.7 ~ 0.9 | 강함 | 뚜렷한 선형 패턴 |
| 0.4 ~ 0.7 | 중간 | 선형 경향 있음 |
| 0.2 ~ 0.4 | 약함 | 약간의 관계 |
| 0.0 ~ 0.2 | 매우 약함 | 거의 관계 없음 |

**이 프로젝트**: r=0.768 → **강한 양의 상관**

**면접 질문**: "상관관계와 인과관계의 차이는?"
> **A**: "상관관계는 두 변수가 함께 변한다는 뜻이고,
> 인과관계는 한 변수가 다른 변수를 일으킨다는 뜻입니다.
>
> 예시:
>   아이스크림 판매량 ↑ ↔ 익사 사고 ↑ (상관관계 O, 인과관계 X)
>   → 숨은 변수: 기온 (더우면 둘 다 증가)
>
> 이 프로젝트:
>   CCTV ↑ ↔ 범죄 ↑ (상관관계 O)
>   CCTV → 범죄 감소? (인과관계 X, 오히려 역방향)
>
> 인과관계 입증 조건:
>   1. X와 Y 상관관계 존재 ✅
>   2. X가 Y보다 시간적으로 앞섬 ❌ (단면 데이터라 모름)
>   3. 제3의 변수가 둘 다 일으키지 않음 ❌ (인구밀도 등)
>
> → 시계열 데이터로 확장하면 2번 조건 해결 가능"

### 10.2 다중 선형 회귀

**모델**:
```
Y = β₀ + β₁X₁ + β₂X₂ + ε

여기서:
  Y = 범죄_per_1000 (종속변수)
  X₁ = CCTV_per_1000 (독립변수 1)
  X₂ = 인구밀도 (독립변수 2)
  β₀ = 절편
  β₁, β₂ = 회귀계수
  ε = 오차항
```

**Python 구현**:
```python
from statsmodels.api import OLS, add_constant

X = df[['CCTV_per_1000', '인구밀도']]
X = add_constant(X)  # 절편 추가
y = df['범죄_per_1000']

model = OLS(y, X).fit()
print(model.summary())
```

**결과 해석**:
```
R-squared: 0.590
→ 독립변수들이 종속변수 변동의 59% 설명

Adj. R-squared: 0.551
→ 변수 개수 보정한 설명력

F-statistic: 15.23 (p<0.001)
→ 모델 전체가 통계적으로 유의함

회귀계수:
  CCTV_per_1000: 0.45 (p=0.002)
  → CCTV 1대/천명 증가 → 범죄 0.45건/천명 증가

  인구밀도: 0.003 (p=0.045)
  → 인구밀도 높을수록 범죄 증가
```

**면접 질문**: "R²와 Adjusted R²의 차이는?"
> **A**: "R²는 설명력을 나타내지만, 독립변수를 무조건 추가하면 높아지는 문제가 있습니다.
> Adjusted R²는 변수 개수로 페널티를 주어 이를 보정합니다.
>
> 공식:
>   Adj. R² = 1 - [(1-R²)(n-1)/(n-k-1)]
>   여기서 n=샘플 수, k=독립변수 개수
>
> 이 프로젝트:
>   R² = 0.590
>   Adj. R² = 0.551
>   → 차이가 크지 않아 변수가 적절함을 의미"

> 표준화 함수를 작성하여 72%에서 96% 매칭률로 개선했습니다.

**Q4. 가장 보람찼던 점은?**
> A: 역인과관계를 발견한 것입니다. 일반적 상식과 반대 결과가 나와
> 정책 해석의 오류 가능성을 제시할 수 있었습니다.

**Q5. 이 프로젝트로 무엇을 배웠나요?**
> A: 데이터 분석의 전체 파이프라인(수집→정제→분석→시각화→배포)을 경험했고,
> 데이터 품질 관리와 통계적 사고의 중요성을 배웠습니다.

### 데이터 처리 (10개)

**Q6. Pandas를 선택한 이유는?**
> A: (3.1 참고) 테이블 데이터 처리 표준 라이브러리이고,
> SQL과 유사한 문법으로 병합, 집계가 간편합니다.

**Q7. merge와 concat의 차이는?**
> A: merge는 공통 키 기준 JOIN, concat은 단순 이어붙이기입니다.

**Q8. 결측값을 0으로 채운 이유는?**
> A: 범죄 데이터에서 '-'는 '발생하지 않음'을 의미해서 0이 적절했습니다.

**Q9. 인구 정규화를 왜 했나요?**
> A: (문제 2 참고) 자치구별 인구가 5배 차이나서 절대값 비교 시 왜곡됩니다.

**Q10. 왜 1000을 곱하나요?**
> A: 0.0155보다 15.5대/천명이 이해하기 쉽기 때문입니다.

**Q11. 데이터 병합 시 how='inner'를 선택한 이유는?**
> A: CCTV, 범죄, 인구 모두 있는 자치구만 분석하기 위함입니다.
> outer join은 결측값이 많아져 분석이 부정확해집니다.

**Q12. 데이터 품질을 어떻게 검증했나요?**
> A: describe()로 통계량 확인, isnull().sum()으로 결측값 확인,
> boxplot으로 이상치 탐지를 수행했습니다.

**Q13. 자치구 표준화 함수를 정규표현식으로 안 쓴 이유는?**
> A: (문제 3 참고) 패턴이 단순해서 endswith()로 충분했고,
> 코드 가독성이 더 좋습니다.

**Q14. 중랑구는 왜 제외했나요?**
> A: 원천 데이터가 없어서 분석 대상에서 제외하고 보고서에 명시했습니다.

**Q15. 파생 변수를 왜 만들었나요?**
> A: CCTV_per_1000 같은 파생 변수로 인구 편향을 제거하고,
> CCTV효과범죄로 분석 타겟을 명확히 했습니다.

### 통계 분석 (15개)

**Q16. 상관계수 0.768은 무엇을 의미하나요?**
> A: (10.1 참고) 강한 양의 상관관계로, CCTV 많을수록 범죄 많음을 뜻합니다.

**Q17. p-value가 왜 중요한가요?**
> A: (10.1 참고) p<0.05면 우연이 아니라 통계적으로 유의합니다.

**Q18. 상관관계와 인과관계의 차이는?**
> A: (10.1 참고) 상관은 함께 변하는 것, 인과는 원인-결과 관계입니다.

**Q19. 역인과관계를 어떻게 발견했나요?**
> A: (결과 2, 문제 1 참고) CCTV 상위 5개 = 범죄 상위 5개로
> 60% 일치하여 역인과관계 가능성을 확인했습니다.

**Q20. R²=0.590은 좋은 건가요?**
> A: (결과 1 참고) 사회과학에서 0.5 이상이면 준수한 수준입니다.
> 인간 행동은 복잡해서 한 변수로 설명하기 어렵습니다.

**Q21. R²와 Adjusted R²의 차이는?**
> A: (10.2 참고) Adj. R²는 변수 개수로 페널티를 주어 보정합니다.

**Q22. 다중공선성은 확인했나요?**
> A: VIF를 계산하여 모든 변수가 VIF<10임을 확인했습니다.

**Q23. 회귀 가정 검토는 어떻게 했나요?**
> A: VIF(다중공선성), Q-Q Plot(정규성), 잔차 산점도(등분산성)를 확인했습니다.

**Q24. 왜 Pearson 상관을 썼나요?**
> A: 두 변수가 모두 연속형이고 선형 관계를 가정할 수 있어서입니다.
> Spearman은 순위 상관으로 비선형에 적합하지만, 이 경우는 선형이 적절했습니다.

**Q25. CV=0.419는 무엇을 의미하나요?**
> A: (결과 3 참고) 변동계수 41.9%로 자치구 간 차이가 크다는 뜻입니다.
> 균등 정책보다 맞춤형 정책이 필요합니다.

**Q26. Z-score를 왜 사용했나요?**
> A: 위험도를 표준화하여 자치구 간 상대적 위치를 비교하기 위함입니다.

**Q27. 시계열 분석을 하지 않은 이유는?**
> A: 단면 데이터만 확보 가능했습니다. 향후 연도별 데이터 확보 시
> 패널 회귀로 인과관계를 분석할 계획입니다.

**Q28. 샘플 수 24개로 통계 분석이 유효한가요?**
> A: t-검정은 n≥30 권장이지만, 상관분석은 n≥20이면 가능합니다.
> 또한 전수 조사(25개 자치구 중 24개)라 샘플링 오차가 없습니다.

**Q29. 유의수준 0.05를 선택한 이유는?**
> A: 일반적으로 사회과학에서 5% 유의수준을 사용합니다.
> 1종 오류(False Positive) 확률 5%를 허용하는 수준입니다.

**Q30. 단측 검정 vs 양측 검정?**
> A: 양측 검정을 사용했습니다.
> CCTV와 범죄가 양의 상관인지 음의 상관인지 사전에 모르기 때문입니다.

### 4사분면 분류 (10개)

**Q31. 중앙값을 사용한 이유는?**
> A: (6.2 Step 3 참고) 평균은 극단값에 영향 받지만,
> 중앙값은 정확히 50%씩 분할하여 강건합니다.

**Q32. Q2를 왜 우선순위로 두나요?**
> A: (6.3 참고) CCTV 적은데 범죄 많아서 추가 설치 효과가 클 것으로 예상됩니다.

**Q33. Q1은 왜 우선순위가 아닌가요?**
> A: 이미 CCTV가 많은데도 범죄가 많아서
> CCTV 외에 다른 대책(경찰 순찰, 조명)이 필요합니다.

**Q34. 4개 지역만 우선순위로 충분한가요?**
> A: (6.3 참고) 예산 제약 시 집중 투자가 균등 배분보다 효율적입니다.

**Q35. 비즈니스 임팩트는?**
> A: (6.3 참고) 집중 투자 시 범죄 예방 효과 30% 향상 예상됩니다.

**Q36. 분면 이동을 어떻게 추적하나요?**
> A: (기능 3 참고) 연도별 데이터로 Q2→Q4 이동 여부를 모니터링할 수 있습니다.

**Q37. 다른 분류 방법을 고려했나요?**
> A: K-Means 클러스터링도 고려했지만,
> 중앙값 기준이 해석이 더 쉽고 정책 설명에 유리했습니다.

**Q38. Q3은 어떻게 관리하나요?**
> A: 현상 유지하면서 범죄 증가 추세를 모니터링합니다.
> 예산 절약 대상입니다.

**Q39. Q4는 모범 사례인가요?**
> A: CCTV 많고 범죄 적어서 효과적으로 보이지만,
> 인과관계가 아닐 수 있어 추가 검증이 필요합니다.

**Q40. 3개 이상 변수로 확장 가능한가요?**
> A: 가능합니다. 조명, 경찰 순찰을 추가하면
> 3차원 분류나 다차원 스코어링으로 확장할 수 있습니다.
